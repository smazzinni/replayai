    59→const introduction = `# Introduction
    60→
    61→ReplayAI is a DVR for AI agent workflows. It records every agent execution — every LLM call, tool call, retrieval, and decision — as a fully replayable session you can scrub, diff, and export as a deterministic regression test.
    62→
    63→## Why it exists
    64→
    65→AI agents are non-deterministic. The failure you saw in production at 3am almost never reproduces when you re-run the same prompt locally. Existing logging gives you text dumps, but you can't *re-run* the exact conditions that broke.
    66→
    67→ReplayAI fixes this by capturing the full execution trace and letting you replay it with every external call mocked from the recording — identical bytes, every time, at zero cost.
    68→
    69→## What you get
    70→
    71→- **Session recording** — a single decorator wraps your agent and captures inputs, outputs, timing, tokens, and costs.
    72→- **Deterministic replay** — re-run any recorded session with tool and RAG responses served from the recording. Free and 100% reproducible.
    73→- **Visual timeline** — scrub through every step like a video. See call ordering, durations, and failures at a glance.
    74→- **Diff view** — compare two sessions side-by-side to find exactly where behavior diverged.
    75→- **Export to test** — one click turns a failing session into a pytest or jest regression test with mocks auto-extracted.
    76→
    77→## How it works
    78→
    79→\`\`\`text
    80→┌──────────────┐     record      ┌──────────────┐     store     ┌────────────┐
    81→│  Your agent  │ ──────────────▶ │  replayai    │ ────────────▶ │  Session   │
    82→│  (LangChain, │   trace()       │  tracer      │   JSON        │  store     │
    83→│   CrewAI…)   │                 └──────────────┘               └────────────┘
    84→└──────────────┘                                                        │
    85→                                                                        │ replay
    86→                                                                        ▼
    87→                                                              ┌──────────────────┐
    88→                                                              │  Dashboard / CLI  │
    89→                                                              │  · timeline       │
    90→                                                              │  · diff           │
    91→                                                              │  · export → test  │
    92→                                                              └──────────────────┘
    93→\`\`\`
    94→
    95→The tracer is a thin instrumentation layer — no inference, no model calls of its own. It just observes and records. During replay, it serves recorded responses for every external call (tools, retrieval, and optionally LLMs) so the agent re-executes deterministically.

   258→const concepts = `# Core Concepts
   259→
   260→ReplayAI has four primitives: **projects**, **sessions**, **steps**, and **replay**. Understanding these four is all you need.
   261→
   262→## Projects
   263→
   264→A project is a logical grouping of sessions — typically one per agent deployment. Projects let you scope the dashboard, filter sessions, and manage access in the cloud product.
   265→
   266→\`\`\`python
   267→# Pin a run to a specific project
   268→@trace("support-agent-v3", project="support-agent")
   269→def handle_support_ticket(message): ...
   270→\`\`\`
   271→
   272→If you omit \`project\`, the SDK uses the \`REPLAYAI_PROJECT\` env var, then falls back to the first project in your store.
   273→
   274→## Sessions
   275→
   276→A session is a single recorded execution of an agent — one user message in, one final response out (or a failure). A session contains:
   277→
   278→| Field | Description |
   279→| --- | --- |
   280→| \`id\` | Stable identifier (\`ses_xxxx\`), used in share links and tests |
   281→| \`status\` | \`success\`, \`failed\`, or \`running\` |
   282→| \`startedAt\` | When the run began |
   283→| \`durationMs\` | Wall-clock duration |
   284→| \`tokenTotal\` | Sum of input + output tokens across all LLM calls |
   285→| \`costUsd\` | Estimated cost, derived from per-model rates |
   286→| \`tags\` | Free-form labels for filtering (\`production\`, \`canary\`, …) |
   287→| \`steps\` | Ordered list of steps (see below) |
   288→
   289→A session is **immutable once it's flushed**. To "edit" one, you delete and re-record — which is exactly what you want for a debugging artifact. Immutability is what makes replays trustworthy.
   290→
   291→## Steps
   292→
   293→A step is a single observable action the agent took. There are five types:
   294→
   295→| Type | Color | What it represents | Replay behavior |
   296→| --- | --- | --- | --- |
   297→| \`llm_call\` | emerald | A call to a language model | Replayed verbatim by default; opt-in to re-invoke live |
   298→| \`tool_call\` | sky | A function/tool the agent invoked (e.g. \`issue_refund\`) | Always mocked from recording |
   299→| \`retrieval\` | violet | A RAG retrieval (vector search, keyword, hybrid) | Always mocked from recording |
   300→| \`decision\` | amber | A routing/branching decision the agent made | Replayed verbatim |
   301→| \`error\` | rose | A failure (tool error, exception, guardrail trip) | Reproduced deterministically |
   302→
   303→Each step records its name, offset from session start, duration, status, model (for LLM calls), token counts, input, and output. That's it — no internal state, no black box.
   304→
   305→The key insight: **only \`llm_call\` steps are non-deterministic in the original run**. Everything else (tools, retrieval, decisions) is deterministic given the same inputs. So during replay, we mock the deterministic external calls and serve the recorded LLM responses — giving you a fully reproducible execution.
   306→
   307→## Replay
   308→
   309→Replay re-runs a recorded session under the exact conditions that produced it. There are two modes:
   310→
   311→### Mocked LLM (default, free, deterministic)
   312→
   313→Every step — tools, retrieval, *and* LLM responses — is served from the recording. The agent re-executes but every external call short-circuits to the recorded bytes. Output is byte-identical to the original. Costs nothing. Use this for:
   314→
   315→- Debugging a specific failure
   316→- Regression tests in CI
   317→- Sharing a reproduction with a teammate
   318→
   319→### Live LLM (opt-in, paid, non-deterministic)
   320→
   321→Tools and retrieval are still mocked, but LLM calls are re-invoked against the real model with the recorded input. Useful for:
   322→
   323→- Testing whether a prompt change produces different output
   324→- Catching model drift before deploy
   325→- A/B comparing two model versions on identical inputs
   326→
   327→You toggle this per-run in the dashboard, or with the \`--live-llm\` flag in the CLI.
   328→
   329→## What gets recorded (and what doesn't)
   330→
   331→**Recorded:** every LLM call's full prompt + completion, every tool call's args + result, every retrieval's query + returned chunks, all timing, all token counts, all errors.
   332→
   333→**Not recorded:** secrets in environment variables (the SDK redacts known secret patterns), file contents not passed through the agent, network traffic not mediated by the agent.
   334→
   335→> **Note:** The SDK redacts values matching common secret patterns (\`sk-\`, \`Bearer \`, \`password=\`, etc.) before writing to the store. You can add custom redaction patterns via [\`REPLAYAI_REDACT_PATTERNS\`](?view=developers&doc=sdk-config).
   336→`;

   338→const sdkPython = `# Python SDK
   339→
   340→The Python SDK is the primary way to instrument agents. It's framework-agnostic at its core — the framework extras just pre-wire callbacks.
   341→
   342→## \`trace()\` — the only function you need
   343→
   344→\`trace()\` is both a decorator and a context manager. It starts a session, records every step inside it, and flushes on exit.
   345→
   346→\`\`\`python
   347→from replayai import trace
   348→
   349→# As a decorator
   350→@trace("support-agent-v3", project="support-agent", tags=["production"])
   351→def handle_support_ticket(message: str) -> str:
   352→    ...
   353→
   354→# As a context manager
   355→with trace("research-agent-v2", tags=["batch"]):
   356→    result = crew.kickoff(inputs={"topic": "AI observability pricing"})
   357→\`\`\`
   358→
   359→### Signature
   360→
   361→\`\`\`python
   362→def trace(
   363→    name: str,
   364→    *,
   365→    project: str | None = None,       # project slug or id; defaults to env
   366→    tags: list[str] | None = None,     # free-form labels
   367→    framework: str = "Custom",         # metadata only
   368→    started_at: datetime | None = None,# defaults to now()
   369→) -> TraceContext
   370→\`\`\`
   371→
   372→The returned \`TraceContext\` is also a context manager — exiting it flushes the session to the store and broadcasts a \`session:created\` event to any connected dashboards.
   373→
   374→## Manual step recording
   375→
   376→Normally you don't record steps manually — the framework integrations do it for you. But for custom agents or when you want explicit control:
   377→
   378→\`\`\`python
   379→from replayai import trace, record_step
   380→
   381→with trace("custom-agent") as ctx:
   382→    # Record an LLM call manually
   383→    record_step(
   384→        type="llm_call",
   385→        name="classify_intent",
   386→        model="gpt-4o-mini",
   387→        tokens_in=312,
   388→        tokens_out=24,
   389→        input="User: refund please",
   390→        output="intent: billing_dispute",
   391→        status="success",
   392→    )
   393→
   394→    # Record a tool call
   395→    record_step(
   396→        type="tool_call",
   397→        name="issue_refund",
   398→        input='{"charge_id": "ch_002"}',
   399→        output='{"refund_id": "ref_3391"}',
   400→        status="success",
   401→    )
   402→
   403→    # Record an error
   404→    record_step(
   405→        type="error",
   406→        name="refund_blocked",
   407→        input="charge_id=ch_002",
   408→        output="ERROR 403: requires approval_id",
   409→        status="failed",
   410→    )
   411→\`\`\`
   412→
   413→Steps are timestamped automatically relative to session start. The \`t\` (offset) and \`durationMs\` fields are inferred unless you pass them explicitly.
   414→
   415→## \`ReplaySession\` — for tests and CLI replay
   416→
   417→\`ReplaySession\` loads a recorded session and re-runs it under mocked (or live) conditions. This is what the exported tests use.
   418→
   419→\`\`\`python
   420→from replayai import ReplaySession
   421→
   422→replay = ReplaySession("ses_8fa1")
   423→
   424→# Register mocks explicitly (optional — the recording is used by default)
   425→replay.mock("issue_refund", '{"refund_id":"ref_3391","status":"succeeded"}')
   426→
   427→# Run the agent under recorded conditions
   428→with replay.trace() as trace:
   429→    result = replay.run(agent="support-agent-v3", framework="LangChain")
   430→
   431→assert trace.step_count == 8
   432→assert trace.status == "failed"
   433→\`\`\`
   434→
   435→### \`ReplaySession\` API
   436→
   437→\`\`\`python
   438→class ReplaySession:
   439→    def __init__(self, session_id: str, *, live_llm: bool = False): ...
   440→
   441→    def mock(self, fn_name: str, response: str | dict) -> None:
   442→        """Override the recorded response for a tool/retrieval call."""
   443→
   444→    def run(self, *, agent: str, framework: str = "Custom") -> Any:
   445→        """Re-execute the agent. Tools/RAG are mocked; LLM is replayed unless live_llm=True."""
   446→
   447→    def trace(self) -> TraceContext:
   448→        """Context manager yielding a Trace with .step_count, .status, .steps[]."""
   449→
   450→    def export(self, lang: str = "pytest") -> str:
   451→        """Generate a test file as a string. lang='pytest' | 'jest'."""
   452→\`\`\`

   454→## Async support
   455→
   456→The SDK is fully async-compatible. Use \`atrace\` and \`arecord_step\`:
   457→
   458→\`\`\`python
   459→import asyncio
   460→from replayai import atrace, arecord_step
   461→
   462→@atrace("async-agent")
   463→async def handle(message: str) -> str:
   464→    intent = await classify(message)
   465→    await arecord_step(type="llm_call", name="classify", ...)
   466→    return intent

   471→## Error handling
   472→
   473→The tracer never raises into your agent code. If recording fails (disk full, network error to cloud store), it logs a warning and continues — your agent's correctness never depends on ReplayAI.
   474→
   475→You can opt into strict mode if you want recording failures to surface:
   476→
   477→\`\`\`python
   478→import replayai
   479→replayai.strict_mode = True  # raises RecordingError on failure

   691→
   692→In production you may not want to record every run (cost, storage, latency). Set \`REPLAYAI_SAMPLE_RATE\` to a fraction:
   693→

   707→## Cost estimation
   708→
   709→The SDK estimates cost from token counts and per-model rates. Built-in rates cover OpenAI and Anthropic models; for others, add entries to \`[cost_rates]\` in your config:

   793→## The common LangChain failure mode
   794→
   795→The most common LangChain agent bug: the agent **hallucinates a tool argument**. Here's the pattern ReplayAI catches:
   796→
   797→\`\`\`text
   798→Step 5 (LLM): "I'll process the refund of $24.00."
   799→Step 6 (Tool): issue_refund() → ERROR 403: requires approval_id
   800→Step 7 (LLM): "I'll retry with approval_id='mgr_auto_2024'"  ← HALLUCINATED
   801→Step 8 (Tool): issue_refund(retry) → ERROR 401: invalid token
   802→\`\`\`
   803→
   804→Without ReplayAI, you'd see the final error in your logs and have no idea the agent fabricated an approval token. With the timeline, step 7's output makes the hallucination obvious — and you can replay it to test your fix.

  1128→## A helper to reduce boilerplate
  1129→
  1130→Manual \`record_step\` calls get verbose. Wrap your LLM and tool calls:
  1131→
  1132→\`\`\`python
  1133→from contextlib import contextmanager
  1134→import time
  1135→from replayai import record_step
  1136→
  1137→@contextmanager
  1138→def step(type_: str, name: str, **kwargs):
  1139→    """Context manager that records a step with automatic timing."""
  1140→    start = time.perf_counter()
  1141→    try:
  1142→        yield
  1143→    except Exception as e:
  1144→        record_step(
  1145→            type=type_,
  1146→            name=name,
  1147→            status="failed",
  1148→            duration_ms=int((time.perf_counter() - start) * 1000),
  1149→            output=f"ERROR: {e}",
  1150→            **kwargs,
  1151→        )
  1152→        raise
  1153→    else:
  1154→        record_step(
  1155→            type=type_,
  1156→            name=name,
  1157→            status="success",
  1158→            duration_ms=int((time.perf_counter() - start) * 1000),
  1159→            **kwargs,
  1160→        )
  1161→
  1162→# Usage
  1163→with step("llm_call", "Classify intent",
  1164→          model="gpt-4o-mini", input=message,
  1165→          tokens_in=resp.usage.prompt_tokens, tokens_out=resp.usage.completion_tokens,
  1166→          output=intent):
  1167→    resp = client.chat.completions.create(...)
  1168→    intent = resp.choices[0].message.content
  1169→\`\`\`
  1170→
  1171→## Wrapping the OpenAI client directly
  1172→
  1173→For a zero-boilerplate approach, monkey-patch (or wrap) the OpenAI client so every call is recorded:
  1174→
  1175→\`\`\`python
  1176→import openai
  1177→from replayai import record_step
  1178→
  1179→class TracedOpenAI:
  1180→    def __init__(self):
  1181→        self._client = openai.OpenAI()
  1182→
  1183→    @property
  1184→    def chat(self):
  1185→        return TracedChat(self._client.chat)
  1186→
  1187→class TracedChat:
  1188→    def __init__(self, chat):
  1189→        self.completions = TracedCompletions(chat.completions)
  1190→
  1191→class TracedCompletions:
  1192→    def __init__(self, completions):
  1193→        self._completions = completions
  1194→
  1195→    def create(self, **kwargs):
  1196→        resp = self._completions.create(**kwargs)
  1197→        record_step(
  1198→            type="llm_call",
  1199→            name=kwargs.get("model", "unknown") + " call",
  1200→            model=kwargs.get("model"),
  1201→            tokens_in=resp.usage.prompt_tokens,
  1202→            tokens_out=resp.usage.completion_tokens,
  1203→            input=str(kwargs.get("messages", "")),
  1204→            output=resp.choices[0].message.content,
  1205→            status="success",
  1206→        )
  1207→        return resp
  1208→
  1209→client = TracedOpenAI()
  1210→\`\`\`
  1211→
  1212→Now every \`client.chat.completions.create()\` is recorded automatically — no manual steps needed. This is essentially what the framework extras do, generalized.
